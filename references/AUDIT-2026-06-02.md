# Tilted Web App Audit - 2026-06-02

## Scope

This is a second-pass audit of the current `main` iteration of Tilted. It reviews player
flow, motion controls, deck authoring, accessibility, browser storage, offline behavior,
static hosting, Docker deployment, GHCR publishing, CI, and the next useful product
features.

The first audit from 2026-06-01 produced a strong round of improvements. The current app
now has more than 70 built-in decks with at least 50 cards each, randomized rounds,
permission-denied fallback, persistent sound and vibration preferences, custom-deck
recovery, offline shell caching, hardened Nginx headers, pinned CI actions and base
images, public GHCR publishing, and a read-only capability-dropped container example.

The app is in good shape for real play. The next iteration should fix a few reliability
edges, then invest in features that make Tilted feel like a complete multiplayer game
rather than a polished single-round prototype.

## Executive Recommendation

Prioritize the next work in this order:

1. Fix the editor exit and cross-deck undo bugs.
2. Add a short input lock so one card cannot be recorded twice during feedback.
3. Reduce sensor-event rerenders and make timer behavior resilient to browser throttling.
4. Build a local multiplayer session mode with team rotation and cumulative scoring.
5. Add motion diagnostics, wake-lock support, richer deck discovery, and round history.
6. Improve deck sharing, install/update UX, and static-host security documentation.

## Priority Reliability Findings

### 1. Invalid deferred deck edits can leave the editor and remain playable

**Priority:** High
**Area:** Data integrity / custom decks
**Evidence:** `src/components/DeckEditor.tsx:101-115`, `src/components/DeckEditor.tsx:249-255`

Typing in a deck name, category, prompt, or hint updates the parent `customDecks` state
immediately, then defers LocalStorage validation and persistence for 350 ms. When the
player taps **Done**, `flushDecks()` attempts to persist the pending state, but the
handler calls `onBack()` immediately even if validation fails.

For example, a player can erase a required prompt, tap **Done**, and return to the home
screen with an invalid custom deck still present in React state. It may be playable
during the current session but disappear after refresh because the invalid edit was
never stored.

**Recommendation:** Make `flushDecks()` return success or failure and block navigation
when validation fails. Prefer keeping a validated saved copy plus a separate editor
draft so invalid text remains editable without leaking into the playable library.

### 2. Card-deletion undo can restore a card into the wrong deck

**Priority:** High
**Area:** Data integrity / custom decks
**Evidence:** `src/components/DeckEditor.tsx:50`, `src/components/DeckEditor.tsx:174-196`

The undo state stores only `{ card, index }`. If a player deletes a card from deck A,
switches to deck B, and taps **Undo**, the deleted card is inserted into deck B.

**Recommendation:** Store the original `deckId` with the deleted card. Reset the undo
notice when the selected deck changes, or restore only into the captured source deck.
Add a regression test for switching decks between delete and undo.

### 3. Rapid touch or keyboard input can record more than one outcome during feedback

**Priority:** High
**Area:** Game correctness
**Evidence:** `src/components/GameScreen.tsx:87-133`

Motion controls have cooldown and neutral rearming, but touch buttons and keyboard
shortcuts do not have an equivalent feedback lock. A fast double tap or repeated arrow
key can advance multiple cards while the 500 ms full-screen feedback layer is visible.
Two events delivered before React rerenders can also record the same card more than once.

**Recommendation:** Add an `inputLockedRef` that is set before recording an outcome and
released when feedback ends. Ignore all input sources during that short transition.
Add tests for rapid touch clicks and rapid key presses.

### 4. Device-orientation samples rerender the app at sensor frequency

**Priority:** Medium
**Area:** Efficiency / mobile battery use
**Evidence:** `src/hooks/useMotionControls.ts:278-280`

Every accepted `deviceorientation` event updates `currentSample` state. The current UI
does not render `currentSample`, but this still causes the top-level `App` component to
rerender as often as the sensor reports samples. On a phone, that can be dozens of
rerenders per second while a round is active.

**Recommendation:** Keep the hot-path sample in `currentSampleRef`. Only publish sample
state on a throttled interval when a diagnostics screen is visible. Keep action,
permission, and status updates reactive because those change the UI.

### 5. The round timer can drift when the browser throttles intervals

**Priority:** Medium
**Area:** Game correctness
**Evidence:** `src/hooks/useTimer.ts:24-43`

The timer subtracts one second for each delivered interval callback. Mobile browsers can
delay intervals when the page is briefly backgrounded, the operating system is busy, or
the screen state changes. A delayed callback therefore makes a nominal 60-second round
last longer than 60 seconds.

**Recommendation:** Track an absolute deadline and derive remaining time from the
current clock. Recalculate on each interval and on `visibilitychange`. Pause should
preserve the remaining duration rather than continue advancing the deadline.

### 6. A failed calibration silently becomes a button-only game

**Priority:** Medium
**Area:** Motion UX
**Evidence:** `src/App.tsx:102-107`, `src/hooks/useMotionControls.ts:232-252`

`finishCalibration()` correctly returns `false` when no orientation sample arrives, but
`finishCountdown()` ignores the return value and starts the game. The fallback buttons
appear because motion is not calibrated, which keeps gameplay available, but the player
does not learn why motion controls vanished.

**Recommendation:** Carry a one-line fallback notice into the game or return to setup
with **Continue with buttons** and **Try motion again** actions. Keep the current
non-blocking behavior.

### 7. Browser-storage failures are not consistently contained

**Priority:** Medium
**Area:** Robustness / privacy modes
**Evidence:** `src/services/deckStorage.ts:127-128`, `src/services/preferences.ts:8-15`

Custom-deck writes are wrapped with a useful storage-full error, but initial reads and
preference reads/writes call LocalStorage directly. Browsers can throw when storage is
disabled by policy, embedded browsing context restrictions, or privacy modes.

**Recommendation:** Add a small safe-storage wrapper that catches read, write, and
remove failures. Continue with in-memory defaults and show a non-blocking message that
custom data will not survive refresh.

### 8. Container publishing is not gated on the test workflow

**Priority:** Medium
**Area:** Release safety / supply chain
**Evidence:** `.github/workflows/ci.yml`, `.github/workflows/publish-container.yml`

`Verify app` and `Publish container` run independently after a push to `main`. The
Docker build catches TypeScript and production-build failures, but it does not run the
Vitest suite. A behavioral regression can therefore be published if it compiles.

**Recommendation:** Run the verification commands in the publish job before pushing,
or trigger publication only after the `Verify app` workflow succeeds for the same
commit. Add Docker updates to Dependabot so pinned Node and Nginx digests stay current.

### 9. Service-worker updates need explicit lifecycle handling

**Priority:** Medium
**Area:** Offline reliability / PWA UX
**Evidence:** `public/sw.js:1-65`, `src/main.tsx:10-16`

Offline caching is a useful addition, but the cache version is static and the app does
not surface update state. Navigation responses are stored under `index.html` without
checking `response.ok`, so a transient proxy error could become the offline fallback.
Runtime-cached hashed assets also remain until the cache name changes.

**Recommendation:** Cache navigation HTML only when `response.ok`, bump or generate a
revisioned cache name during releases, delete obsolete runtime entries, and show an
**Update available - Reload** banner when a new service worker takes control. Add
offline and update-path tests.

### 10. The About modal needs complete keyboard focus management

**Priority:** Low
**Area:** Accessibility
**Evidence:** `src/components/AboutModal.tsx:34-124`

The About modal has dialog semantics and Escape handling, but it does not move focus
inside the modal, trap focus while open, restore focus on close, or lock background
scroll.

**Recommendation:** Use a reusable modal primitive or add focus capture, focus restore,
tab containment, and body-scroll locking. Preserve backdrop click and Escape behavior.

## Feature Roadmap

### 1. Add local multiplayer sessions

**Priority:** High
**Why:** This is the largest gap between the current app and the intended Heads Up-style
party experience.

The app currently plays one isolated round and shows one result. Add a **Party Mode** or
**Team Game** flow:

- Create two or more teams or player names.
- Choose the number of rounds or a target score.
- Rotate the phone holder automatically after each round.
- Keep a cumulative scoreboard.
- Show the next player or team before the forehead countdown.
- Preserve the current **Quick Round** path for classroom use.

Keep the first version local-only and stored in memory. A backend is not needed for a
shared-room experience because the phone already acts as the game controller.

### 2. Add persistent round history and useful statistics

**Priority:** High
**Why:** The current results screen is good immediately after a round, but the initial
product plan included round history and the app does not preserve it.

Store recent rounds locally with:

- Timestamp, deck, duration, mode, team or player, correct count, and pass count.
- Ordered card outcomes rather than separate correct/pass arrays only.
- A compact recent-rounds screen.
- Optional clear-history and export actions.
- Per-deck best score and recent-score summary.

For classrooms, a downloadable CSV summary would be useful without introducing
accounts or student data storage on a server.

### 3. Add motion sensitivity presets and a motion test screen

**Priority:** High
**Why:** Motion behavior is the most device-dependent part of the app and has already
needed tuning.

Add a small **Test Motion Controls** flow from round setup:

- Show whether permission, orientation samples, forehead placement, calibration, and
  neutral rearming are working.
- Display simple visual states such as **Neutral**, **Tilt farther down**, **Correct
  detected**, and **Pass detected**.
- Offer sensitivity presets such as **Standard**, **Gentle**, and **Deliberate**.
- Keep the current 40-degree threshold as the default.
- Persist the selected preset locally.

Throttle any diagnostic visualization so it does not reintroduce sensor-frequency
rerenders during normal play.

### 4. Add wake-lock, fullscreen, and best-effort landscape locking

**Priority:** High
**Why:** A forehead game benefits from keeping the screen bright and stable throughout
the round.

Use progressive enhancement:

- Request a screen wake lock when a round starts and release it after the round.
- Reacquire the wake lock after tab visibility changes when appropriate.
- Offer a one-tap fullscreen action where the browser supports it.
- Attempt `screen.orientation.lock("landscape")` only where allowed, such as installed
  PWAs or fullscreen experiences.
- Keep the current portrait gate because browsers do not reliably expose rotation-lock
  state and cannot always force rotation.

### 5. Improve deck discovery for a 70-plus-deck library

**Priority:** Medium
**Why:** Category scrolling works, but the larger library now needs faster discovery.

Add:

- Search by deck name and description.
- Favorite decks.
- Recently played decks.
- A **Surprise Me** random-deck action.
- An optional mixed-deck builder that combines selected categories.
- A custom-deck filter alongside built-in categories.

The remaining player-facing `N decks` category label in
`src/components/DeckSelectScreen.tsx:62-65` can also be removed if the cleaner,
less-technical deck-selection presentation is preferred.

### 6. Use the existing difficulty and category metadata during play

**Priority:** Medium
**Why:** Cards already carry `difficulty` and `category`, but round setup does not use
them.

Add optional round filters:

- Easy, standard, hard, or mixed difficulty.
- Subcategory selection for review decks.
- A **no repeats** versus **cycle deck if needed** choice.
- Optional pass limit for competitive rounds.

Cycling is worth considering because a very small custom deck or a fast team can exhaust
all available cards before the timer ends.

### 7. Expand custom-deck authoring and sharing

**Priority:** Medium
**Why:** The editor is already useful, but the next additions would make it practical
for educators and party hosts.

Add:

- A deck-description field. The model supports `description`, but the editor does not.
- Whole-library backup and restore, not only one-deck export.
- CSV import and export for spreadsheet-oriented users.
- Duplicate-prompt warnings against cards already in the selected deck.
- Card reorder, bulk delete, and bulk category assignment.
- Share via the Web Share API when supported.
- Optional QR-code or URL-based deck import for moving a deck between phones without
  accounts.

For larger decks, paginate, virtualize, or collapse card rows. The current editor can
render up to 500 cards with four form controls per card.

### 8. Add PWA install and update affordances

**Priority:** Medium
**Why:** Tilted already has a manifest and service worker, but players must discover
installation behavior on their own.

Add:

- An install prompt where `beforeinstallprompt` is supported.
- iOS-specific **Add to Home Screen** instructions.
- A lightweight offline indicator.
- A reload banner when a new version is ready.
- Manifest fields such as `description`, `start_url`, and `scope` after confirming
  subdirectory deployments.

### 9. Add classroom presentation options

**Priority:** Medium
**Why:** Tilted is useful in classrooms as well as parties.

Consider:

- A teacher-controlled **Review Mode** with no tilt requirement.
- Projector-friendly results and scoreboard layouts.
- A setting that suppresses entertainment-themed decks.
- Deck tags for subject, age range, and classroom-safe status.
- An optional end-of-round export that avoids collecting student names by default.

### 10. Make self-hosted metadata and hardening easier to configure

**Priority:** Low
**Why:** The public GHCR image is easy to run, but deployment-specific metadata is fixed
at image-build time.

Add:

- Runtime metadata injection for the Docker image, or a documented custom-build path in
  the Unraid template.
- Netlify `_headers`, Vercel header config, or equivalent examples for users who deploy
  static files without the bundled Nginx server.
- Reverse-proxy guidance for HSTS after HTTPS is confirmed.
- Docker Dependabot updates for the pinned base images.

The current defaults intentionally point at the public repository and canonical cover
asset. That is valid, but self-hosted instances should have an easy path to advertise
their own public URL.

## Security Posture

The current security baseline is appropriate for a small public client-only app:

- The app has no backend, account system, cookies, or server-side data store.
- React escapes card text before rendering.
- Imported decks are length-limited, shape-validated, and capped at 500 cards.
- Malformed custom-deck data is salvaged where possible and preserved for recovery.
- The Docker runtime is unprivileged, read-only, capability-dropped, and configured
  with `no-new-privileges`.
- Nginx sends CSP, Permissions Policy, Referrer Policy, MIME-sniffing protection, and
  frame-denial headers.
- GitHub Actions and base container images are pinned to immutable revisions.
- The repository and GHCR image are public as intended.
- Anonymous GHCR manifest access succeeds.
- `npm audit --json` reports zero known vulnerabilities.

Remaining security work is mainly operational:

- Gate GHCR publication on tests.
- Add Docker ecosystem updates to Dependabot.
- Keep HTTPS termination and HSTS configuration correct at the reverse proxy.
- Provide equivalent security-header examples for static hosts.
- Continue treating custom decks as local browser data rather than a place to store
  sensitive student information.
